语音对话系统

LLM 语音对话
发布日期:   2024-11-05
更新日期:   2025-03-09

1. 产品应用

语音只是交互方式，带来的挑战是语音的理解、回复语音的自然生动、低延迟可打断；核心仍然是基于 llm 的领域内知识的专业度、内容生产、情感价值，优化回复自然度、应用诉求的生成能力。

具体应用场景下，级联系统、端到端系统的选择 ……

2. 系统结构

2.1. turn take 判断

  • 必要性：从语音人机交互体验，需解放双手，不论级联系统、turn based 模型、全双工模型。

2.1.1. 方案 1：VAD + turn take 判断模型

  • 缺点
    • 数据集构建：结合语义区分 query 结束、打断、back-channel、背景人声等；准入的非语音段。
      • 实现方案：类似 Freeze-Omni，LLM head 添加分类层，prefill，同时区分继续接收语音、打断、开始回复、语音结束不需要打断（丢弃该语音片段？重新 prefill）。
      • 注意：处理系统回声，含打断时的重叠人声。如：前端 AEC（webrtc）、数据增强。
    • 错误累积。
      • 优化方案：保召回，LLM 数据增强，非查询可不回复。
    • 能力扩展：支持纯音乐输入，哭声、咳嗽等声音事件的情感支持。
    • VAD 区分短暂停顿、语音结束导致的延迟。<= 问题 1 的优化方案可缓解
      • 优化方案：如 Step-Audio，若用户短暂暂停，生成推测性响应，若用户继续说话，则丢弃推测性响应，约 40% 的推测性响应被执行，响应延迟减少了约 500ms，用多余的计算降低延迟。
  • 注意：流式，依赖的未来帧数，影响延迟。
  • 示例：MinMo、VITA、Freeze-Omni、mini-omni (“StopOmni”)

2.1.2. 方案2：LLM 始终接收用户输入

  • 缺点
    • 计算量：用户静音时段不必要的大模型推理。
    • 首包延迟：（非编解码）音频合成通常需要一段时间的文本/音频 token 来保障连贯性，音频 chunk 交错格式下引入额外的 (audio_chunk)/2、moshi 格式下引入 audio_chunk 的首包延迟。
  • 示例：Moshi、OmniFlatten

2.2. 音频表示

2.2.1. 方案1：离散表示

  • 缺点：相较于连续表示，用于 ASR 等音频理解任务性能有损，特别是噪声、口音数据。

  • 示例

    • CosyVoice、GLM-4-Voice ASR encoder 中间层量化；Step-Audio 采用 Paraformer、SenseVoice-Large encoder 双码本；
    • Baichuan-Audio: 8层 RVQ，频谱重建、音频理解多任务训练；Moshi 采用 split RVQ，将自监督音频模型的语义表示蒸馏至 codec，对抗损失训练。

2.2.2. 方案2：audio 输入采用连续表示，输出采用离散表示

  • 输入、输出均可非流式计算。
  • 示例：Minmo、MiniCPM-o 2.6 等。Ola 将 Whisper、BEATS（音乐 encoder）表示在 channel 维拼接。

2.2.3. 方案3：连续表示

  • 音频生成：如字节 DiTAR

2.3. 离散表示

2.3.1. 方案1：semantic token

  • 缺点：音乐、背景声音事件生成

2.3.2. 方案2：semantic token + acoustic token，split RVQ

  • 缺点：多音色支持

2.4. 音频生成

2.4.1. 方案1：LLM 预测文本 + TTS 模型

  • 缺点：LLM 预测文本中需要包含语种、方言、情感、声音事件等 tag，训练数据需要细致的标注，难以泛化到未见 tag。
  • 示例：Step-Audio 生成文本包含指定格式的 tag，支持语种、方言、语速、情感控制，正常语音、RAP、哼唱区分；说话人利用提示音频作为 TTS LM 对话历史、flow matching prompt。

2.4.2. 方案2：利用 LLM 表示 + TTS

  • 优点：音频生成由另外的模块完成，降低了主 LLM 建模难度、推理延迟（小 LM 生成长音频序列更快），但利用 LLM 的表示实现指令跟随的、富有表现力的音频生成。
  • 示例

    • MinMo TTS 模块的输入为用户输入、响应文本，音频/文本 表示，及对应的 LLM 最后一层隐藏状态，在特征维拼接。通过多阶段对齐训练，文本 LLM 原始能力几乎无损。

    • Moshi: Depth Transformer 输入为 text/audio embedding + Linear(last_hidden_state)，自回归预测 audio token。

    • Baichuan-Audio: LLM 预测音频隐层表示 + Depth Transformer + 8个分类头。

    • MiniCPM-o 2.6 LLM 先预测1个 speech embedding，系统 prompt 中支持加入音频进行音色克隆。

2.4.3. 方案3：LLM 直接预测 audio token

  • 缺点：相较于仅预测文本，同时预测文本、音频，性能下降。故 mini-omni 提出并行解码策略。

    * S->S：输出交错的文本-音频，评价文本。S->T：输出仅文本。

3. 效果优化

3.1. 智商

  • MinMo 利用大规模训练数据、多阶段对齐训练，与 ASR + 文本 LLM 性能相当。

  • 相较于 S -> T，S-> S 在知识问答任务上存在约 10 个点的准确率下降。从 MinMo 的结果看，由于答案中的专业词汇对 TTS 有挑战、ASR 性能影响评价。

3.2. 延迟

  • 流式音频 token 提取：通常需要向后看一段时间。

  • 自然对话平均延迟约为 230ms。Moshi 理论延迟 160ms，实际延迟 200ms。

  • MinMo 基于 cosyvoice 2，理论延迟 600ms，实际约为 800ms。

  • LLM、flow matching 推理加速。

3.3. 对话自然度

LLM 改写文本对话数据集回复，使符合口语对话的长度、风格。

3.4. 情感对话

LUCY 通过合成数据微调来增强响应的情感控制、自然度，并最先生成响应情感 token。

3.5. 生成语音的细粒度控制

  • Step-Audio 提出了高质量 TTS 训练数据合成 pipeline：通过收集各语种、方言 native speaker 数据，利用中间 checkpoint 快速扩展 native speaker、目标说话人 多语种、方言数据；SFT 训练得到 Audio-Edit 模型将音频情感、风格化。
  • OpenOmni 进行了情感语音生成 DPO 训练，采用 CosyVoice 分别生成无条件语音、情感条件语音作为正负样本对。

3.6. 指令跟随、对话自然度、安全性等

  • RL：Step-Audio 特别地将音频清晰、回复包含 “I didn’t hear clearly” 等存在 deaf hacking 现象的样本作为负样本。

3.7. 工具调用

  • LUCY 基于函数调用数据集（如 WebPCM、DevOpsEval），保留适合语音交互的，如谷歌搜索、天气搜索、新闻搜索等，采用 LLM 改写回复。并在开始添加简洁的确认，以补偿函数调用延迟。

3.8. 多轮对话长上下文

  • 多轮对话历史仅采用文本，转写用户语音，回复文本采用生成文本，如 Step-Audio、SLAM-Omni、LUCY（用户语音未转写，回复仅保留文本）。但 SLAM-Omni 指出“在某些情况下，历史上下文保留非语言信息对于保持对话的连贯性和深度至关重要”。
  • SLAM-Omni 将输入的连续音频表示 5 帧拼接、每一步预测 1个文本 token + 3个音频 token，显著减小了序列长度。
  • SpeechGPT 2.0-preview 将输入的音频表示多帧拼接、每一步预测 1个文本 token + 多解码头的自回归语言模型生成多帧 RVQ token。

3.9. 歌声生成

  • Step-Audio 采用多个 TTS 模型，通过响应文本中的特殊 tag 确定调用哪一个模型。

3.10. 其它经验

  • 预训练：模态交错数据有助于知识迁移。可采用 text-to-audiotoken 模型扩充。
  • SFT 数据：
    • 声音指令：支持生成语音的自然语言细粒度控制，如方言、情感、角色扮演、RAP、唱歌等。
    • 特定 case，增强鲁棒性。
      • 用户指令语义不明或音质较差，模型要求用户重说，如 Sorry, I'm not sure what you're referring to.。
      • 用户提问包含事实性错误，模型回答 "No" 并纠正。
      • 自我认知。
  • 模型输出
    • 有文本引导时语音生成质量更好。

3.11. 注意点

  • 无意义查询的合理回复，避免幻觉。
