Minmo

LLM 语音对话 论文
发布日期:   2025-02-19
更新日期:   2025-02-19
  • Chen Q, Chen Y, Chen Y, et al. Minmo: A multimodal large language model for seamless voice interaction[J]. arXiv preprint arXiv:2501.06282, 2025.
  • https://funaudiollm.github.io/minmo/ 暂未开源，提供 demo 视频
  • 作者：来自阿里通义
  • 创新点
    • 采用对齐多模态大模型方案，利用大规模训练数据、多阶段对齐训练，在多个任务上取得了 SOTA 效果，且文本 LLM 原始能力几乎无损。存疑：无 Qwen2.5-7B-instruct 对话能力对比；知识问答任务上，相较于S2T，S2S 准确率折损均在 10% 以上。
    • 流式语音生成：采用 Output Projector（利用 LLM 隐藏层表示）+ Voice Token LM（自回归地生成语音 token）+ 流式 ( flow matching + vocoder )，实现了低延迟、自然、指令跟随的语音生成。
    • 实现全双工预测模块，利用 LLM 的语义理解能力，实现了低延迟的全双工交互，理论延迟约为 600ms，实际约为 800ms。
  • 可借鉴点
    • 音频输入采用连续表示，声学 encoder 采用 ASR、副语言、音频事件等任务大规模有监督训练；
    • 音频生成由小 LM 完成，降低 LLM 建模难度、推理延迟（小 LM 生成长音频序列更快），但利用 LLM 隐层表示实现指令跟随的、富有表现力的音频生成。
    • 全双工交互：利用小模型持续接收用户音频输入，判断何时调用 LLM 进行响应、提供给 LLM 的音频片段起止。本文
      • 未详细介绍服务运行流程。
      • 保留 VAD
        • 优点：可避免用户静音时 LLM 的推理 cost；
        • 缺点：需要保留用户查询中的音频、音乐段；无法响应用户非主动语音查询时的声音事件，如哭闹。
      • 采用额外的模块，利用 LLM 的语义理解能力，区分用户语音输入为查询还是 back channel。
    • 数据
      • 语音到文本对齐，Qwen-Audio -> Minmo，仅保留对话相关的任务。
      • SFT 数据：采用 LLM、TTS 合成风格可控的语音对话。
    • 多阶段训练：逐模块对齐、任务由易到难。

1. 系统结构

模块 结构 作用 参数量
Voice Encoder SenseVoice-Large encoder 提取连续音频表示 ~ 636M
Input Projector 2 Transformer 层 + 1 CNN 层 维度对齐、2倍下采样 ~ 170M
LLM Qwen2.5-7B-instruct 生成文本回复 7B
Output Projector 1 线性层 维度对齐 ~ 6M
Voice Token LM CosyVoice 2 LM 自回归地生成语音 token ~ 370M
Full Duplex Predictor 1 Transformer 层 + 1 线性分类层 输入 LLM 隐藏状态，输出：speak or not。存疑：LLM 自回归生成文本的过程中，如何监听用户语音输入？类似 VITA 额外部署吗？ ~ 18M
  • 参数量总约 8B。
  • Token2wav Synthesizer: CosyVoice 2 flow matching（chunk-aware，支持 chunk=15）+ vocoder。

1.1. Output Projector

  • 输入：query embedding + response embedding，在序列维度拼接。
  • query embedding：用户输入 embedding + LLM 最后一层隐藏状态，在特征维度拼接。保留用户指令信息。
  • response embedding：5个采样的文本 token 的 embedding + LLM 最后一层隐藏状态。
    • 采样的文本：相较于隐藏状态，可使音频更一致。

1.2. Voice Token LM

  • 输入文本表示、输出音频 token 5:15 交错，实现流式生成，降低语音回复延迟。若文本结束、语音未结束，插入 <turn of speech> token，后续仅语音。生成 <end of speech> token 则结束。

2. 数据

* 说话人分析：性别、年龄、说话人计数、说话人 identification、多说话人识别、目标说话人识别。

合计约 140wh 音频。

  • 语音-文本
    • 约 120wh，ChatML 格式，自然语言任务指令。
  • 语音-语音
    • 文本聊天数据（Alpaca、ShareGPT）TTS，用户语音零样本 in-context 合成。
    • 挑选 ASR 数据，LLM 生成回复、TTS。增强对真实音频的鲁棒性。缺点：只能合成单轮对话？
    • 风格可控的对话：LLM 生成风格可控的文本多轮对话。录制了小规模、多样化、富有表现力的语料库作为回复语音 TTS 的提示。
    • TTS：CosyVoice，助手语音采用特定说话人 2h 数据微调的 SFT 模型合成。

3. 训练

stage 更新的模块
语音到文本对齐 先仅训练 Input Projector，避免随机初始化的 Projector 在训练初期对 encoder 产生大的梯度；
再更新 Voice Encoder + Input Projector；
最后 LoRA 微调 LLM 增强指令跟随能力，各任务数据量均衡，见论文图4。存疑：前面的阶段指令较少主要学习对齐？后来丰富了指令？
文本到语音对齐 先仅训练 Output Projector；
再更新 Output Projector + Voice Token LM。
语音到语音对齐 Output Projector + Voice Token LM
全双工交互对齐 Full Duplex Predictor

5. 局限性

  • 遵循多样化的指令的能力，如语种、任务。
    • 优化方案：更多高质量的文本、LLM 全面更新。
  • 存在长尾发音错误
    • 原因：LLM 一对多 token（一个 token 多个发音，如多语种共用 token、多音字？）、输出文本中的某些特殊符号无法转换为语音。
    • 优化方案：数据扩展。
  • 仍需要 AEC、VAD。
