愿，彼此相伴，走过流年

Moshi

LLM 多模态 论文
发布日期:   2024-10-10
更新日期:   2024-10-12
  • Défossez A, Mazaré L, Orsini M, et al. Moshi: a speech-text foundation model for real-time dialogue[J].
  • https://arxiv.org/abs/2410.00037
  • https://github.com/kyutai-labs/moshi
  • 创新点
    • 第一个 multi-stream 全双工、实时流式口语对话框架。理论延迟 160ms，实际延迟 200ms（自然对话平均延迟约为 230ms）。支持的上下文约为 5min。
    • semantic-acoustic audio codec，将自监督音频模型的语义表示蒸馏至 codec，并采用 split RVQ 权衡语言区分性及重建质量；因果、低帧率、低比特率，支持实时流式。
    • 相较于语义到声学 token 生成，首先生成时间对齐的文本 token，指导音频 token 的生成；audio codec 引入语义 token；以及在 RQ-Transformer 结构中引入声学延迟，显著提高了生成语音的语言质量。
    • 通过改变不同类型 token 之间的延迟，支持流式 ASR、TTS。

1. 系统结构

1.1. LLM - Helium

  • 7B 参数。
  • tokenizer：SentencePiece，32000，主要为英语，数值全部拆分为单个数字，byte-backoff。
  • RoPE。
  • RMS 归一化：attention block、feed-forward block、线性输出层 的输入。
  • FFN：采用 GLU 及 SiLU 门控函数。

1.2. audio tokenizer - audio codec - Mimi

1.2.1. 结构

  • SeaNet。输入 24kHz 的采样点，通过卷积得到 12.5帧/s、512 维的表示。decoder 采用类似的结构还原为 24kHz 的音频。因果卷积，支持流式。
  • 量化前后分别添加 Transformer 模块，8层，10s 上下文（论文中两处有出入），因果 mask。
    • 为了稳定训练，采用 LayerScale，初始化为对角线值 0.01。
    • 收益见表 3。

1.2.2. 量化

  • 8 码本，码本大小 2048，256 维。
  • split RVQ：语义量化 + 7层传统声学 RVQ。
    • 类似 SpeechTokenizer，从自监督模型 WavLM 中蒸馏语义信息。
    • 音频下采样到 16kHz --> WavLM（50Hz，1024维）--> 平均池化（s=4，k=8，12.5Hz。非因果：对性能很重要，仅用于训练）。
    • RVQ 语义表示线性投影到 1024 维。计算与 WavLM embedding 的 cosine 距离损失。
  • 采用 quantizer dropout，bitrate scalability。
  • 训练时，各序列仅 50% 的时刻被量化，收益见表 3。

1.2.3. 训练损失

对比了 重建 + 对抗 损失、仅对抗损失（特征 + 鉴别）。采用后者重建质量（与参考音频的声学相似度）客观指标显著下降，但主观评价显著提高，见表 4。

1.3. RQ-Transformer

\[ \begin{aligned} z_s &=\operatorname{Tr}_{\operatorname{Temp}}\left(V_0, \ldots, V_{s-1}\right) \in \mathbb{R}^d \\ l_{s, 1} &=\operatorname{Lin}\left(z_s\right) \in \mathbb{R}^{N_1} \\ l_{s, k} &=\operatorname{Tr}_{\text {Depth }}\left(z_s, V_{s, 1}, \ldots, V_{s, k-1}\right) \in \mathbb{R}^{N_k} \end{aligned} \]

  • Depth Transformer 中，对于不同的子序列 \(k\)，线性层参数不一样。

  • 声学延迟

    生成更稳定。同时生成的声学 token 与语义 token 的依赖减弱，同一时刻子序列间的依赖越复杂，需要越强大的模型来建模。语义、声学 token 间的关系由更大的 Temporal Transformer 建模，而不是小的 Depth Transformer。（存疑：\(z_s\) 表示的语义信息、声学信息不同步，表示难度更大？）

    \[\begin{cases} V_{s, 1}=A_{s, 1} & \\ V_{s, q}=A_{s-\tau, q} & \text { if } \quad s \geq \tau+1, q>1 \\ V_{s, q}=0 & \text { if } \quad s<\tau+1, q>1 \end{cases}\]

1.4. Inner Monologue

label：whisper ASR，带词级时间戳。添加 special token \(\text {PAD}\)、\(\text {EPAD}\)。

\[ \text{初始化} \quad \mathrm{Wt} \leftarrow \mathrm{PAD} \quad \forall \mathrm{t} \\ \left\{\begin{array}{ll} W_{t_i+j} & \leftarrow w_{i, j} \quad \quad \forall j \leq n_i \\ W_{t_i-1} & \leftarrow \mathrm{EPAD} \quad \text{若不覆盖文本 token} \end{array} \right. \]

其中，\(t_i\) 为第 i 个单词对应的起始索引，\(w_{i, j}\) 为该单词对应的文本 tokens。若第一帧为语音，插入 \(\text {EPAD}\)。可以强制采样 \(\text {EPAD}\) 使 Moshi 立即开始说话。未根据单词结束时刻区分静音段？

*用户音频数据增强：
50% 音量缩放 [-24, 15] dB；

2.1. 合成指令数据集

在 Open Hermes 文本指令数据集、真实对话转写上微调 Helium，用于生成对话。

2.1.1. TTS 训练

训练流式、multi-stream TTS，用于合成指令微调数据集。

  • audio 预训练共享 Moshi 的。
  • post-training
    • 数据集：两人 自然 + 脚本 对话，高质量、分声道录制，转写。170h。
    • 文本、音频延迟：2s。

2.1.2. 对话生成

  • 一般知识：给定 Wikipedia 段落、StackExchange 帖子、或者关于它自己的信息，作为 {{context}}，生成关于 {{title}} 的对话 {{摘要}}，进一步生成对话。
  • 声音指令：单轮，指示模型用 {{特定的声音}} 说关于 {{实体}} 的内容（句子、诗歌等）。
  • 角色扮演：系统为 {{特定情绪或说话风格}} 的 {{角色}}，生成 {{场景}}，进一步生成对话。Moshi 的文本可以包含说话风格、动作等，如 MOSHI: (relieved) *raises fists* Hooray! To the moon and beyond!。
  • 安全性、数学、闲聊。
  • 特定 case，增强鲁棒性。
    • 用户指令包含拼写错误，模型要求用户重说，如 Sorry, I'm not sure what you're referring to.。
    • 用户提问包含事实性错误，模型回答 "No" 并纠正。

2.2. loss

\[ L(V, l)=\frac{1}{S} \sum_{s=1}^S\left(\mathrm{CE}\left(l_{s, 1}, V_{s, 1}\right)+\frac{1}{\sum_{k=2}^K \alpha_k} \sum_{k=2}^K \alpha_k \mathrm{CE}\left(l_{s, k}, V_{s, k}\right)\right) \] 其中，\(k=1\) 对应文本 token，语义 token \(\alpha_k=100\)，声学 token \(\alpha_k=1\)。

  • 音频 artifacts 检测
    • 在各时间步 \(s\)，对各 token level \(k\)，分别计算窗 \(C=64\)（约 4.5s）内 token 的熵 \(H_s^k=H\left(A_{s-C: s, k}\right)\)。根据非重叠窗 \(\omega=64\) 内的熵 \(H_{s: s+\omega}^k\) 检测是否存在 artifacts。
    • 重复文本：文本 token 的熵比较平坦。对 \(H_{s: s+\omega}^0\) 拟合线性回归，斜率小于阈值 \(\eta_{\text {flat }}=10^{-3}\)。
    • 静音：文本 token 为 \(\text {PAD}\)，\(H_{s: s+\omega}^0=0\)。音频 token 熵较低 \(\operatorname{median}_{k>1, s}\left(H_{s: s+\omega}^k\right) \leq \eta_{\text{audio_silence}} =2\) 。
    • 背景噪声：文本 token 为 \(\text {PAD}\)，音频 token 熵超过上述阈值。
    • Gibberish：胡言乱语，文本 token 熵很高 \(H_{s: s+\omega}^0>\eta_{\text {gibberish }}=3.5\)。量化至 2bit 时出现频率较高。
    • 噪声：文本 token 非 \(\text {PAD}\)，\(\operatorname{SD}_{k>1, s}\left(H_{s: s+\omega}^k\right) > \eta_{\text{noise}} =0.6\)。

4. 其它

  • 相较于级联系统
    • 低延迟；
    • 保留了非语言信息，如情感、声音事件；
    • 支持打断、backchanneling（不打断的反馈，如 OK、I see）。
  • 纯语音模型虽然可以从语音中学习语言结构，如词汇、句法、语义，但事实知识、推理能力很差，因此通常结合文本模型的知识、推理能力和生成模型的生成能力。
  • 声学 token 对精细的音频细节进行建模，用于高质量重建；语义 token 用于表征连贯、可理解的音频，可以从自监督音频模型中提取。
  • audio codec 不是 idempotent 的：从 tokens 生成音频、重新编码，tokens 大概率不一致。token 对时间偏移不鲁棒。特别是仅采用感知对抗目标训练的 audio codec。详见表 16。
  • 个人思考 Q&A
    • 为什么需要语义 token：仅重建目标训练的 codec 语言区分性较差，用于表征用户语音输入时难以表示语言内容。（生成时，先生成文本 token，语义 token 的作用还大吗？）
